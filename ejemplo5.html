🧩 Cómo funciona el fetch con useEffect

Qué sucede cada vez que el usuario escribe un nombre de usuario:

usuario cambia
→
useEffect corre
→
cargando = true
→
fetch a GitHub API
Si la respuesta es exitosa:
res.ok = true
→
setDatos(json)
→
muestra tarjeta
Si el usuario no existe:
res.ok = false
→
throw Error
→
muestra error
Dispara el efecto
useEffect
Petición HTTP
Éxito
Error

Este componente maneja 4 estados. Cada uno tiene una responsabilidad distinta:

usuario — string
El texto que el usuario escribe en el input. Inicia en ''. Cada cambio dispara el efecto porque está en el array de dependencias.
datos — object | null
La respuesta de la API de GitHub. Inicia en null. Cuando llega la respuesta, guarda el objeto con login, avatar_url, public_repos, etc.
cargando — boolean
Indica si la petición está en curso. Se pone en true al iniciar el fetch y vuelve a false cuando termina — exitosa o con error.
error — string | null
Mensaje de error si el usuario no existe o falla la red. Inicia en null. Se limpia a null al iniciar cada nueva búsqueda.
Este patrón de 4 estados (datos, cargando, error, más el valor que dispara la búsqueda) es el más común en aplicaciones React que consumen APIs. Vale la pena memorizarlo.

07
Importar useState y useEffect
import { useState, useEffect } from 'react';
08
Declarar los 4 estados

Este componente necesita 4 estados. Declara cada uno con su valor inicial.

const [usuario,  setUsuario]  = useState('');
const [datos,    setDatos]    = useState(null);
const [cargando, setCargando] = useState(false);
const [error,    setError]    = useState(null);
datos inicia en null porque todavía no hay ninguna respuesta de la API. El JSX usará esto para decidir si mostrar la tarjeta o no.
09
Escribir el useEffect con fetch

El efecto se dispara cada vez que usuario cambia. Si el input está vacío, sale sin hacer nada.

useEffect(() => {

  if (!usuario.trim()) return;  // si está vacío, no buscar

  setCargando(true);
  setError(null);               // limpiar error anterior

  fetch(`https://api.github.com/users/${usuario}`)
    .then((res) => {
      if (!res.ok) throw new Error('Usuario no encontrado');
      return res.json();
    })
    .then((json) => {
      setDatos(json);
      setCargando(false);
    })
    .catch((err) => {
      setError(err.message);
      setDatos(null);
      setCargando(false);
    });

}, [usuario]);
[usuario] en el array de dependencias significa que el efecto corre cada vez que el input cambia. A diferencia del reloj donde usamos [], aquí sí necesitamos que el efecto reaccione a un valor.
res.ok es true si el status HTTP es 200-299. Si el usuario no existe, GitHub devuelve un status 404 — res.ok será false y lanzamos el error manualmente con throw.
⚠️ fetch no lanza un error automáticamente cuando el status es 404 o 500 — solo falla si hay un problema de red. Por eso siempre hay que verificar res.ok manualmente.
10
Escribir el JSX

Muestra el input, y de forma condicional: el spinner de carga, el mensaje de error o la tarjeta del usuario.

return (
  <div className="buscador-contenedor">
    <h1 className="buscador-titulo">Buscador de GitHub</h1>

    <input
      type="text"
      placeholder="Escribe un usuario de GitHub..."
      value={usuario}
      onChange={(e) => setUsuario(e.target.value)}
      className="buscador-input"
    />

    {cargando && <p className="buscador-mensaje">Buscando...</p>}

    {error && <p className="buscador-error">{error}</p>}

    {datos && !cargando && (
      <div className="perfil-card">
        <img
          src={datos.avatar_url}
          alt={datos.login}
          className="perfil-avatar"
        />
        <div className="perfil-info">
          <h2 className="perfil-nombre">{datos.name || datos.login}</h2>
          <p className="perfil-usuario">@{datos.login}</p>
          <p className="perfil-repos">
            Repositorios públicos: <strong>{datos.public_repos}</strong>
          </p>
          {datos.bio && <p className="perfil-bio">{datos.bio}</p>}
        </div>
      </div>
    )}

  </div>
);
datos.name || datos.login — algunos usuarios de GitHub no tienen nombre configurado. Si name es null, mostramos login como fallback.
{datos.bio && ...} — la bio es opcional. Si el usuario no tiene bio, no renderiza nada. Es el mismo patrón de renderizado condicional con &&.

🎯 Reto extra
Buscar solo al presionar Enter o un botón

Actualmente la búsqueda se dispara con cada letra que escribe el usuario. Modifica el componente para que solo busque cuando el usuario presione Enter o haga clic en un botón "Buscar". Requiere separar el estado del input del estado que dispara el efecto.

const [inputValue, setInputValue] = useState('');
const [usuario,    setUsuario]    = useState('');

// El efecto ahora depende de usuario, no de inputValue
useEffect(() => { ... }, [usuario]);

// El input actualiza inputValue, no usuario
<input
  value={inputValue}
  onChange={(e) => setInputValue(e.target.value)}
  onKeyDown={(e) => {
    if (e.key === 'Enter') setUsuario(inputValue);
  }}
/>

<button onClick={() => setUsuario(inputValue)}>
  Buscar
</button>
Ahora hay dos estados separados: inputValue para lo que el usuario escribe y usuario para lo que dispara la búsqueda. El efecto solo corre cuando usuario cambia — es decir, al presionar Enter o el botón.
