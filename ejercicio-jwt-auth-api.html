04
Pruebas con Postman, Bruno o Insomnia

Sustituye el host y el prefijo si tu API usa otra base URL; abajo se usa http://localhost:3000/api.

  • Registro: POST .../auth/registro con cuerpo JSON { "nombre": "...", "email": "...", "password": "123456" } → espera 201 y mensaje de éxito.
  • Login: POST .../auth/login con { "email", "password" } → espera 200 y un objeto con token.
  • Perfil con token: GET .../usuarios/perfil con encabezado Authorization: Bearer <token del login> → espera 200 y id, nombre, email (sin contraseña).
  • Perfil sin token: misma petición sin encabezado → 401 con mensaje de token requerido.
  • Token inválido: Bearer tokeninvalido123 → 401 con mensaje de token inválido o expirado.

05
Flujo en el navegador

CORS (Cross-Origin Resource Sharing): el navegador trata como distintos orígenes dos URLs con distinto protocolo, host o puerto. Si la app React corre en http://localhost:5173 y el API en http://localhost:3000, las peticiones axios son "entre orígenes" y el navegador puede bloquearlas si el backend no responde con las cabeceras que permiten ese origen.

En el backend, en app.js (o el punto de entrada de Express), registra CORS antes de montar rutas y de express.json(), para que todas las respuestas incluyan la política adecuada. Con el paquete cors instalado, una forma habitual en desarrollo es:

Backend — app.js (extracto)
import cors from 'cors';
// ...
app.use(cors());

Si tu backend usa require, equivale a const cors = require('cors'); y luego app.use(cors());.

En producción suele acotarse el origen (por ejemplo solo el dominio del frontend). cors() sin opciones es cómodo para practicar en local.

Si tu proyecto incluye Vite en http://localhost:5173 con pantallas de registro, login y perfil:

  • Registro: al enviar el formulario, deberías terminar en la ruta de login.
  • Login: tras éxito, guarda el token (por ejemplo localStorage.setItem('token', token)) y navega al perfil.
  • En las herramientas de desarrollo, pestaña Aplicación → almacenamiento local, verifica que exista la clave token.
  • Perfil autenticado: debe mostrar nombre y email obtenidos del API.
  • Cierre de sesión: al limpiar el token y salir, al entrar manualmente a /perfil debería redirigir al login si implementaste esa protección en el router.
El frontend debe enviar el mismo encabezado Authorization que probaste en Postman en las peticiones al perfil.
Mapa del flujo
06
De registro a perfil protegido
REGISTRO
  Formulario → POST /api/auth/registro
    → bcrypt.hash(password, 10)
      → Usuario.create({ nombre, email, password: hash })
        → 201 { mensaje } → redirigir a login

LOGIN
  Formulario → POST /api/auth/login
    → Usuario.findOne({ email })
      → bcrypt.compare(password, usuario.password)
        → jwt.sign({ id, email }, process.env.JWT_SECRET, { expiresIn: '7d' })
          → 200 { token } → localStorage.setItem('token', token) → ir a perfil

RUTA PROTEGIDA
  Perfil → GET /api/usuarios/perfil + Authorization: Bearer <token>
    → verificarToken → jwt.verify → req.usuario
      → Usuario.findByPk(req.usuario.id, { attributes: sin password })
        → 200 { id, nombre, email }
Integración en el frontend
07
LoginForm.js: de las credenciales al token

En el frontend del starter, el inicio de sesión suele vivir en LoginForm.js. Ese componente mantiene el correo y la contraseña en estado local, llama al API con axios y usa la URL base definida en import.meta.env.VITE_API_URL (archivo .env del proyecto Vite, por ejemplo VITE_API_URL=http://localhost:3000).

La pieza que conecta el formulario con el backend es handleLogin: envía email y password, espera { token }, guarda el token en localStorage para las rutas protegidas y redirige a /perfil.

LoginForm.js — imports, estado y handleLogin
import { useState } from 'react';
import { Link, useNavigate } from 'react-router-dom';
import axios from 'axios';

export default function LoginForm() {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [error, setError] = useState('');
  const navigate = useNavigate();

  const handleLogin = async () => {
    setError('');
    try {
      // 1. Enviamos las credenciales al backend
      const res = await axios.post(
        import.meta.env.VITE_API_URL + '/api/auth/login',
        { email, password }
      );

      // 2. El backend responde con { token } si las credenciales son correctas
      // 3. Guardamos el token en localStorage para usarlo en rutas protegidas
      localStorage.setItem('token', res.data.token);

      // 4. Redirigimos al usuario a su perfil
      navigate('/perfil');
    } catch (err) {
      setError(err.response?.data?.error || 'Error de conexión');
    }
  };

  // ... return: inputs controlados (value + onChange), botón con onClick={handleLogin}, <Link to="/registro" />, etc.
}

En el return del mismo archivo, los inputs usan el estado (value / onChange), el botón "Entrar" dispara handleLogin con onClick={handleLogin}, y puedes ofrecer el enlace a registro con <Link to="/registro"> de React Router.

RegistryForm.js: alta de usuario y salto al login

El registro suele estar en RegistryForm.js (nombre del componente en el starter). Guarda nombre, email y password en estado, reutiliza axios y la base del API en import.meta.env.VITE_API_URL (aquí como constante apiBase para armar la URL con template string).

handleRegister hace POST /api/auth/registro con el cuerpo esperado por el backend. Si la petición termina bien, no hace falta guardar token todavía: lo habitual es navigate('/login') para que el usuario inicie sesión en la pantalla siguiente.

RegistryForm.js — imports, apiBase, estado y handleRegister
import { useState } from 'react';
import { Link, useNavigate } from 'react-router-dom';
import axios from 'axios';

const apiBase = import.meta.env.VITE_API_URL;

export default function RegistryForm() {
  const [nombre, setNombre] = useState('');
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [error, setError] = useState('');
  const navigate = useNavigate();

  const handleRegister = async () => {
    setError('');
    try {
      await axios.post(`${apiBase}/api/auth/registro`, {
        nombre,
        email,
        password,
      });
      navigate('/login');
    } catch (err) {
      setError(err.response?.data?.error || 'Error de conexión');
    }
  };

  // ... return: campos nombre / email / password, botón con onClick={handleRegister}, <Link to="/login">, etc.
}

En el return, los tres inputs enlazan con nombre, email y password; el botón llama onClick={handleRegister}; el pie suele enlazar a <Link to="/login"> para quien ya tenga cuenta.

09
App.js: proteger la ruta /perfil en el router

En la raíz del frontend (por ejemplo App.jsx o App.js), React Router define las rutas públicas y la de perfil. El punto sensible es /perfil: solo debe mostrarse el componente Perfil si en localStorage ya existe el token que guardaste al hacer login; si no, se redirige a /login con <Navigate to="/login" replace /> para que el usuario no vea una pantalla vacía ni datos sin autenticación.

Esa comprobación es solo en el cliente: mejora la experiencia y evita mostrar la UI del perfil sin sesión, pero la seguridad real sigue en el backend con verificarToken en las peticiones al API.

App.jsx o App.js
import { BrowserRouter, Routes, Route, Navigate } from 'react-router-dom';
import LoginForm from './components/LoginForm.jsx';
import RegistryForm from './components/RegistryForm.jsx';
import Perfil from './pages/Perfil.jsx';
import Home from './pages/Home.jsx';

function App() {
  // Verificamos si existe un token guardado en localStorage
  const token = localStorage.getItem('token');
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/login" element={<LoginForm />} />
        <Route path="/registro" element={<RegistryForm />} />
        <Route
          path="/perfil"
          element={
            // Ruta protegida: si hay token puede entrar, si no regresa al login
            // IMPORTANTE: esta verificación es solo visual en el frontend
            // La verificación real la hace el middleware verificarToken en el backend
            token ? <Perfil /> : <Navigate to="/login" replace />
          }
        />
        <Route path="*" element={<Navigate to="/login" replace />} />
      </Routes>
    </BrowserRouter>
  );
}

export default App;
