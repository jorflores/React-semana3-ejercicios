Ejercicio Node 2

API REST con Express, Sequelize y Neon

Node.js Express Sequelize PostgreSQL Neon ES Modules 60-75 min
Objetivo
00
Migrar el API anterior a una base de datos real

En este ejercicio reutilizarás el proyecto anterior de usuarios, pero eliminarás el arreglo en memoria y guardarás los datos en PostgreSQL usando Neon y Sequelize.

La API conservará las mismas rutas, pero ahora las operaciones GET, POST, PUT y DELETE consultarán una tabla real.

Sequelize es un ORM: permite trabajar con tablas de la base de datos usando modelos y métodos de JavaScript.

Neon y string de conexión
01
Crear cuenta y proyecto en Neon

Si no tienes cuenta, entra a https://neon.tech, crea una cuenta e inicia sesión. Puedes registrarte con correo, GitHub o Google.

Dentro del dashboard crea un proyecto nuevo. Neon generará una base de datos PostgreSQL administrada en la nube.

  • Elige un nombre para el proyecto, por ejemplo api-rest-db.
  • Conserva la base de datos por defecto o crea una llamada usuarios_db.
  • Espera a que Neon termine de crear el proyecto.
02
Copiar el string de conexión

En el panel del proyecto busca la sección de conexión. Neon muestra un string parecido a este:

Ejemplo de formato
postgresql://usuario:password@host.neon.tech/nombre_db?sslmode=require

Copia el string completo. Lo guardarás en un archivo .env para no escribir credenciales directamente en el código.

No publiques tu archivo .env. Ahí estará la contraseña de tu base de datos.

Instalar dependencias

Desde la carpeta del proyecto anterior instala Sequelize, el driver de PostgreSQL, dotenv y nodemon.

Conexión y modelo
07
Crear la conexión con Sequelize

Este archivo lee el string de conexión desde .env y configura SSL, necesario para conectarse a Neon.

config/database.js
import { Sequelize } from 'sequelize';
import dotenv from 'dotenv';

dotenv.config();

const sequelize = new Sequelize(process.env.DATABASE_URL, {
  dialect: 'postgres',
  dialectOptions: {
    ssl: {
      require: true,
      rejectUnauthorized: false
    }
  }
});

export default sequelize;
08
Definir el modelo Usuario

El modelo representa la tabla usuarios. Sequelize puede crearla automáticamente al sincronizar.

Piensa en el modelo como un puente entre JavaScript y PostgreSQL: cada propiedad del objeto será una columna de la tabla, y cada opción describe cómo debe comportarse esa columna.

models/Usuario.js
import { DataTypes } from 'sequelize';
import sequelize from '../config/database.js';

const Usuario = sequelize.define('Usuario', {
  id: {
    type: DataTypes.INTEGER,
    autoIncrement: true,
    primaryKey: true
  },
  nombre: {
    type: DataTypes.STRING,
    allowNull: false
  },
  correo: {
    type: DataTypes.STRING,
    allowNull: false,
    unique: true
  }
}, {
  tableName: 'usuarios',
  timestamps: false
});

export default Usuario;

En este ejemplo, sequelize.define('Usuario', ...) crea un modelo llamado Usuario. Dentro del segundo argumento se definen los campos: id, nombre y correo.

  • DataTypes.INTEGER indica que id será un número entero.
  • autoIncrement: true permite que PostgreSQL genere el siguiente id automáticamente.
  • primaryKey: true marca id como identificador principal de cada usuario.
  • DataTypes.STRING representa texto corto, útil para nombre y correo.
  • allowNull: false obliga a que el campo tenga un valor.
  • unique: true evita que se repita el mismo correo.

En el tercer argumento configuramos opciones del modelo: tableName: 'usuarios' fuerza el nombre real de la tabla y timestamps: false evita que Sequelize agregue automáticamente columnas como createdAt y updatedAt.

Puedes consultar más tipos de datos en la documentación oficial de Sequelize: DataTypes de Sequelize.
Controlador
09
Mover la lógica CRUD al controlador

El controlador reemplaza todas las operaciones sobre el arreglo por consultas con Sequelize. Lo construiremos poco a poco dentro de controllers/usuarioController.js.

Primero importa el modelo Usuario.

controllers/usuarioController.js
import Usuario from '../models/Usuario.js';

const obtenerUsuarios = async (req, res) => {
  const usuarios = await Usuario.findAll();
  res.json(usuarios);
};

La función obtenerUsuarios reemplaza el antiguo res.json(usuarios) del arreglo. Ahora usa Usuario.findAll() para leer todos los registros de la tabla.

async indica que la función trabajará con operaciones asíncronas. await pausa esa función hasta que Sequelize termine la consulta a la base de datos y regrese el resultado. Sin await, intentaríamos responder antes de tener los usuarios.
Agregar debajo de obtenerUsuarios
const obtenerUsuarioPorId = async (req, res) => {
  const usuario = await Usuario.findByPk(req.params.id);

  if (!usuario) {
    return res.status(404).json({ mensaje: 'Usuario no encontrado' });
  }

  res.json(usuario);
};

findByPk busca por llave primaria. En este caso, la llave primaria es el campo id.

Agregar debajo de obtenerUsuarioPorId
const crearUsuario = async (req, res) => {
  const { nombre, correo } = req.body;

  if (!nombre || !correo) {
    return res.status(400).json({ mensaje: 'Nombre y correo son obligatorios' });
  }

  const usuario = await Usuario.create({ nombre, correo });
  res.status(201).json(usuario);
};

Usuario.create() inserta un nuevo registro en PostgreSQL. Ya no necesitas generar el id manualmente porque Sequelize y la base de datos lo hacen por ti.

Agregar debajo de crearUsuario
const actualizarUsuario = async (req, res) => {
  const usuario = await Usuario.findByPk(req.params.id);

  if (!usuario) {
    return res.status(404).json({ mensaje: 'Usuario no encontrado' });
  }

  const { nombre, correo } = req.body;

  await usuario.update({
    nombre: nombre || usuario.nombre,
    correo: correo || usuario.correo
  });

  res.json(usuario);
};

Primero se busca el usuario. Si existe, usuario.update() actualiza el registro real en la base de datos.

Agregar debajo de actualizarUsuario
const eliminarUsuario = async (req, res) => {
  const usuario = await Usuario.findByPk(req.params.id);

  if (!usuario) {
    return res.status(404).json({ mensaje: 'Usuario no encontrado' });
  }

  await usuario.destroy();
  res.json({ mensaje: 'Usuario eliminado correctamente' });
};

usuario.destroy() elimina de PostgreSQL el registro encontrado.

Al final del archivo exporta todas las funciones juntas. Así el archivo queda más fácil de leer y las rutas pueden importarlas por nombre.

Agregar al final de usuarioController.js
export {
  obtenerUsuarios,
  obtenerUsuarioPorId,
  crearUsuario,
  actualizarUsuario,
  eliminarUsuario
};
Estas funciones son async porque las operaciones contra la base de datos tardan más que leer un arreglo local.
Validación de este punto: todavía no se puede probar desde el navegador o Thunder Client. El controlador ya existe, pero falta conectarlo a rutas HTTP y registrar esas rutas en app.js.
Rutas y app
10
Definir las rutas de usuarios

Este archivo conecta cada método HTTP con la función correspondiente del controlador.

Lo hacemos para que app.js no tenga todo el código mezclado. app.js se encargará de arrancar el servidor, mientras que este archivo se encargará solo de decir qué pasa cuando alguien visita una ruta de usuarios.

Un router de Express funciona como una lista de instrucciones: si llega una petición GET /usuarios, ejecuta obtenerUsuarios; si llega POST /usuarios, ejecuta crearUsuario, y así con cada ruta.

routes/usuarioRoutes.js
import express from 'express';
import {
  obtenerUsuarios,
  obtenerUsuarioPorId,
  crearUsuario,
  actualizarUsuario,
  eliminarUsuario
} from '../controllers/usuarioController.js';

const router = express.Router();

router.get('/usuarios', obtenerUsuarios);
router.get('/usuarios/:id', obtenerUsuarioPorId);
router.post('/usuarios', crearUsuario);
router.put('/usuarios/:id', actualizarUsuario);
router.delete('/usuarios/:id', eliminarUsuario);

export default router;

La línea const router = express.Router() crea un mini-enrutador. Después, cada línea router.get, router.post, router.put o router.delete une tres ideas: el método HTTP, la ruta y la función que se debe ejecutar.

Al final usamos export default router para poder importar este conjunto de rutas desde app.js.

Validación de este punto: las rutas ya están definidas, pero aún falta importarlas y usarlas en app.js. La primera prueba real de GET /usuarios se hará después del siguiente paso.
11
Actualizar app.js y sincronizar la base de datos

Ahora modifica el app.js inicial. Agrega la conexión, registra las rutas y sincroniza el modelo con PostgreSQL.

Este archivo será el punto de entrada de la aplicación: aquí se crea el servidor de Express, se activan los middlewares, se conectan las rutas y se arranca el servidor solo si la base de datos responde correctamente.

app.js
import express from 'express';
import dotenv from 'dotenv';
import sequelize from './config/database.js';
import usuarioRoutes from './routes/usuarioRoutes.js';
import './models/Usuario.js';

dotenv.config();

const app = express();
const PORT = process.env.PORT || 3000;

app.use(express.json());
app.use(usuarioRoutes);

app.get('/', (req, res) => {
  res.json({ mensaje: 'API REST de usuarios con Express, Sequelize y Neon' });
});

try {
  await sequelize.authenticate();
  console.log('Conexión a Neon establecida correctamente');

  await sequelize.sync({ alter: true });
  console.log('Modelos sincronizados con la base de datos');

  app.listen(PORT, () => {
    console.log(`Servidor escuchando en http://localhost:${PORT}`);
  });
} catch (error) {
  console.error('Error al conectar con la base de datos:', error);
}

Observa estas líneas clave:

  • import sequelize from './config/database.js' trae la conexión a Neon.
  • import usuarioRoutes from './routes/usuarioRoutes.js' trae el router que definiste en el paso anterior.
  • import './models/Usuario.js' carga el modelo para que Sequelize conozca la tabla usuarios antes de sincronizar.
  • app.use(express.json()) permite que Express lea cuerpos JSON en peticiones POST y PUT.
  • app.use(usuarioRoutes) conecta las rutas de usuarios con la aplicación principal. A partir de ese momento, Express ya sabe responder a /usuarios, /usuarios/:id, etc.

Una forma sencilla de pensarlo es: usuarioRoutes contiene el mapa de rutas, y app.use(usuarioRoutes) pega ese mapa dentro del servidor principal.

El bloque try/catch intenta conectarse a Neon con sequelize.authenticate(). Si la conexión funciona, sequelize.sync() o sequelize.sync({ alter: true }) revisan los modelos antes de iniciar el servidor. Solo después de eso se ejecuta app.listen().

Estas son las diferencias principales:

  • sequelize.sync() crea la tabla si todavía no existe. Si la tabla ya existe, normalmente la deja como está.
  • sequelize.sync({ alter: true }) compara el modelo con la tabla existente e intenta modificar la tabla para que coincida con el modelo. Por ejemplo, puede agregar una columna nueva si la agregaste al modelo.
Referencia importante: sequelize.sync({ force: true }) elimina la tabla existente y la vuelve a crear desde cero. Esto borra los datos guardados, así que no lo uses en esta práctica salvo que quieras reiniciar completamente la tabla.
En esta práctica usamos sync({ alter: true }) porque estamos aprendiendo y puede ser útil ajustar la tabla mientras cambiamos el modelo. En proyectos reales o producción se suelen usar migraciones para controlar mejor los cambios de estructura.
Validación de este punto: guarda los cambios y revisa la terminal. Debes ver los mensajes de conexión a Neon y sincronización de modelos. Después abre http://localhost:3000/usuarios; si la tabla está vacía, debe responder [].
Eliminar el arreglo
12
Quitar el código anterior basado en memoria

Elimina del proyecto anterior el arreglo usuarios y las rutas escritas directamente en server.js.

Ahora la información vive en Neon y la API accede a ella mediante el modelo Usuario.

Después de este cambio, si reinicias el servidor los usuarios ya no se pierden, porque están guardados en PostgreSQL.
